# CLAUDE.md

Project guidance for Claude Code when working in this repo.

## What this repo is

A portfolio-grade Webhook Relay API by Philip Rehberger. Two halves in one
monorepo:

1. **Laravel 13 API + dashboard** at the repo root — REST endpoints, Horizon
   worker pool for fan-out + retries, MySQL persistence, Filament admin.
2. **Next.js 16 docs + marketing site** under `web/` — interactive OpenAPI
   reference (Scalar), conceptual guides, downloadable SDKs.

The whole thing is a sales artifact: it demonstrates that the same person
can deliver a production-shaped API *and* the docs *and* the SDKs *and* the
deploy story. Treat every file as something a prospective client might read.

## Conventions

- **No AI attribution.** No `Co-Authored-By`, no "generated by Claude," no
  AI mentions in commits, code, comments, anything.
- **OpenAPI is the source of truth.** Endpoints in `routes/api.php` conform
  to `openapi/spec.yaml` — not the other way around. CI runs Spectral lint
  + contract tests. When you add a route, add it to the spec first, then
  implement the controller to match.
- **MySQL, not Postgres.** No `jsonb`, no `tsvector`, no partial unique
  indexes. Idempotency dedup uses a separate `idempotency_records` table
  (see the plan for the rationale).
- **ULIDs for all IDs**, stored as `CHAR(26)`.
- **`Authorization: Bearer whk_live_…`** for auth. Keys are hashed at rest;
  the `last_four` and `prefix` are stored plaintext for display.
- **RFC 7807 problem details** for error responses (`application/problem+json`).
- **No premature abstraction.** Three similar repositories beats a generic
  base class. Bug fixes don't need surrounding cleanup. Trust the framework.

## Repo layout

```
/                Laravel root — php artisan works from here
  openapi/       Hand-authored OpenAPI 3.1 spec
  web/           Next.js docs/marketing site (separate deploy target)
  sdks/          Generated SDKs (TS, PHP, Python, Go) — added in Phase 6
  infra/apache/  Canonical Apache vhost files (mirror /etc/apache2/sites-available/)
  infra/cron/    Crontab entries for the EC2 host
  scripts/deploy/  Atomic release-based deploy (cloned from client-portal-laravel)
  .local/        Gitignored secrets (e.g., production MySQL password)
```

## Voice and tone (matches `~/projects/income-ops/CLAUDE.md`)

Marketing copy in the README, the docs site, and any case-study material
follows the same voice as the sales material in `income-ops`:

- Lead with diagnosis, not pitch.
- Concrete numbers over adjectives.
- Em-dashes, short declarative sentences, no hedging.
- No emoji in user-facing copy. No exclamation points.
- "Production-shaped" not "production-grade." (This is a portfolio demo;
  honest framing sells better than overclaiming.)

## Deployment

- **Both halves build locally and rsync to the EC2 host** (54.190.150.0).
  No CI-hosted builds, no Vercel.
- **Apache + Let's Encrypt** for SSL on both subdomains. Vhost edits go in
  `infra/apache/` *and* on the server — keep them in sync.
- **PM2 port 3005** for the Next.js docs site. The reverse-proxy vhost
  forwards there.
- **MySQL `webhook_relay` schema** on the EC2 host. The DB password lives in
  `/var/www/webhook-relay/shared/.env` on the server and in
  `.local/server-secrets.env` locally (both gitignored).

## Things to avoid

- Don't deploy the API until at least the `/v1/healthz` endpoint returns
  200. The deploy script's health check fails the rollout otherwise.
- Don't run `php artisan migrate:fresh` against the production DB without
  asking. There is no separate staging environment yet.
- Don't write SDK code by hand. Always regenerate from
  `openapi/spec.yaml` — the per-language ergonomics wrappers are the only
  hand-written layer.
- Don't add Sentry / observability dependencies until Phase 7+. They're
  out of scope for the demo and add noise to the package manifest.
